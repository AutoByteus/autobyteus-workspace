# Requirements Doc: Iframe Launch Id Contract Refactor

## Status (`Draft`/`Design-ready`/`Refined`)

Design-ready

## Goal / Problem Statement

Refactor the hosted-application iframe bootstrap contract so the existing `launchInstanceId` wording is replaced by an explicitly iframe-scoped bootstrap correlation id, `iframeLaunchId`. The current code uses the id as an ephemeral iframe load/bootstrap attempt discriminator, but its name and current propagation through `ApplicationRequestContext` make it look like AutoByteus models durable application instances or sessions for the normal frontend journey. The target contract must make the durable application identity (`applicationId`) distinct from the iframe bootstrap correlation id.

## Investigation Findings

- **Good-idea assessment: proceed.** Source investigation confirms the requirement matches the current architecture. The id is generated by the web host for each route-visit/reload iframe bootstrap attempt, then used by the web host and hosted frontend SDK to reject stale `ready`/bootstrap messages. It is not a durable backend worker, app-session, run, or business identity.
- Current code duplicates `launchInstanceId` in the URL launch hints, ready payload, bootstrap `launch` object, bootstrap `requestContext`, frontend route headers, REST query/header fallback, server-side request context, tests, docs, samples, generated sample vendors, and a host UI technical-detail label.
- Application backend source in the built-in samples does not use `context.requestContext.launchInstanceId` for behavior. Server code passes the field through or normalizes it, but does not use it for routing, worker lifecycle, storage, run bindings, or app-owned business decisions.
- Because this is a public hosted-iframe/bootstrap contract change, the clean-cut refactor should bump the iframe bootstrap contract version and frontend SDK compatibility version instead of accepting both old and new public shapes.

## Recommendations

- Use `iframeLaunchId` as the single field name across query hints, ready payloads, bootstrap payloads, web host descriptors, SDK startup, tests, docs, and sample app source.
- Replace the old bootstrap `launch { launchInstanceId }` object with one bootstrap-level `iframeLaunchId` field to avoid parallel representations.
- Narrow `ApplicationRequestContext` to `{ applicationId: string }`; do not carry iframe launch identity through normal backend business request context.
- Do not introduce renamed REST headers/query parameters for iframe correlation unless implementation discovers an active business consumer. Current investigation found none, so the design should remove `x-autobyteus-launch-instance-id` and the special `launchInstanceId` query/header handling.
- Bump the iframe contract from v2 to v3 and the frontend SDK compatibility contract from `"2"` to `"3"`. Do not keep v2 validators, aliases, or dual-path compatibility branches for the replaced hosted-iframe behavior.
- Remove host UI technical-detail exposure of the id. Logs and sample diagnostics may mention the id only with iframe-scoped wording.

## Scope Classification (`Small`/`Medium`/`Large`)

Large

The semantic change is straightforward, but the affected surface is broad: shared SDK contracts, frontend SDK, web host components/stores/utils/tests, backend gateway/engine request-context plumbing/tests, manifest compatibility validation, docs, sample app source, generated sample vendors, and generated importable sample packages.

## In-Scope Use Cases

| use_case_id | Use Case | Summary |
| --- | --- | --- |
| UC-001 | Hosted iframe bootstrap | Host creates a fresh iframe bootstrap id, app iframe sends matching ready, host delivers bootstrap only to the matching iframe load. |
| UC-002 | Stale iframe rejection | Host and hosted SDK ignore or fail stale/mismatched ready/bootstrap messages without weakening origin, parent-window, application id, or launch-id checks. |
| UC-003 | Backend business request context | App queries, commands, GraphQL, routes, execution-event handlers, and artifact handlers receive durable `applicationId` context without iframe bootstrap identity. |
| UC-004 | Manifest/package compatibility gating | Packages built for the new frontend SDK contract declare version `"3"`; old version `"2"` is not accepted as a compatibility path for the replaced iframe contract. |
| UC-005 | Developer-facing terminology | Docs, tests, logs, samples, generated vendors, and host UI no longer present iframe correlation as a durable launch instance/session. |

## Out of Scope

- Changing application worker lifecycle ownership; workers remain keyed by durable `applicationId`.
- Stopping the application backend worker on `Exit application`.
- Adding multi-instance application runtime/session semantics.
- Adding new app business identity, app session identity, or run identity fields.
- Maintaining old `launchInstanceId`/`autobyteusLaunchInstanceId` compatibility paths for the in-scope hosted-iframe behavior.
- Redesigning resource setup, run binding, notification streaming, storage, or app-owned business APIs beyond request-context field tightening.

## Constraints / Dependencies

- Keep the iframe trust boundary explicit: origin, parent/source window, `applicationId`, contract version, and `iframeLaunchId` checks must remain visible in the host and hosted SDK.
- The route `:applicationId` remains the authoritative backend boundary for REST gateway calls.
- Backend worker lifecycle remains keyed by durable `applicationId` and is not stopped by `Exit application`.
- Built-in samples commit generated runtime UI/vendor/dist assets; implementation must refresh them after SDK contract changes.
- Dependency installation is absent in the investigated worktree (`node_modules` not present), so validation may require workspace dependency setup before commands can run.

## Assumptions

- `iframeLaunchId` is the preferred final name and is clearer than a nested `iframe: { launchId }` shape because it keeps one exact field name across URL hints, ready payloads, bootstrap payloads, descriptors, and tests.
- No current app backend relies on iframe launch identity for business behavior; current uses are logging, diagnostics, pass-through, or tests.
- Bumping frontend SDK compatibility to `"3"` is acceptable because the user explicitly requested a clean refactor without backward-compatible dual contract paths.
- Generated sample assets should be treated as committed outputs, not ignored artifacts.

## Risks / Open Questions

- Updating generated sample vendors/dist may produce a large diff; reviewers should distinguish generated refresh from hand-authored source changes.
- If an external, non-repo application package currently depends on `launchInstanceId`, it will need to rebuild for frontend SDK contract `"3"`; this is intentional under the clean-cut requirement.
- Some historical documentation may intentionally mention the retired `launchInstanceId` term. Any retained mention must be explicitly labeled historical; otherwise it should be removed.

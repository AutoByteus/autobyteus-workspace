# Requirements

## Status

`Refined`

## Ticket

`codex-runtime-image-input-support`

## Goal / Problem Statement

Ensure Codex runtime supports image input parity with existing autopilot runtime for user-pasted/uploaded images, without regressing text-only behavior.

## Scope Classification

- Classification: `Small`
- Rationale:
  - One codex-runtime input mapper change plus targeted tests.
  - No new public API fields.
  - Existing composer and GraphQL context-file flow already provide image metadata.

## In-Scope Use Cases

- `UC-001` Agent run with runtime `codex_app_server` sends one uploaded image plus text.
- `UC-002` Agent run with runtime `codex_app_server` sends multiple images (mixed local path + uploaded media URL).
- `UC-003` Codex runtime text-only turn still sends successfully.
- `UC-004` Autopilot runtime behavior remains unchanged for image context files.
- `UC-005` Live Codex runtime E2E (`RUN_CODEX_E2E=1`) with real local image fixtures sent via GraphQL context files using both `/rest/files/...` URL form and absolute filesystem-path form succeeds and persists image-bearing user turn input in Codex thread history.

## Out Of Scope

- `OOS-001` New media attachment UX.
- `OOS-002` Video/audio-specific codex input format changes.
- `OOS-003` Team-runtime-specific attachment UX redesign.

## Requirements

| requirement_id | Requirement | Expected Outcome | Use Case IDs |
| --- | --- | --- | --- |
| R-001 | Codex runtime must convert server-local media URLs (`/rest/files/...`) for image context files into `localImage` absolute filesystem paths before `turn/start`. | Codex app-server can load locally uploaded images reliably in desktop/local deployments. | UC-001, UC-002 |
| R-002 | Codex runtime must continue mapping true remote image URLs and data URLs as `image` inputs. | Externally hosted image URLs keep working. | UC-002 |
| R-003 | Codex runtime must preserve existing text-only turn payload behavior. | No text regression. | UC-003 |
| R-004 | Codex runtime URI-to-path conversion must be path-traversal safe and fallback safely when conversion is not valid. | No unsafe filesystem access introduced. | UC-001, UC-002 |
| R-005 | Non-codex runtime behavior must remain unchanged. | Autopilot/runtime adapter parity preserved. | UC-004 |
| R-006 | Repository test suite must include at least one live Codex E2E scenario that exercises image input with a real image fixture. | Real runtime execution proves image-input path works end-to-end in Codex mode. | UC-005 |

## Acceptance Criteria

| acceptance_criteria_id | mapped_requirement_ids | Testable Criteria |
| --- | --- | --- |
| AC-001 | R-001 | Given image context URI `http://<server>/rest/files/images/x.png` (or `/rest/files/images/x.png`), codex mapper emits `{"type":"localImage","path":"<absolute media root>/images/x.png"}`. |
| AC-002 | R-002 | Given `https://example.com/a.png` and `data:image/png;base64,...`, codex mapper emits `{"type":"image","url":...}`. |
| AC-003 | R-003 | Given no context files, codex mapper output remains a single text input item. |
| AC-004 | R-004 | Given malicious path traversal URI (`/rest/files/../../etc/passwd`), codex mapper does not emit unsafe local path. |
| AC-005 | R-005 | Existing runtime adapter tests unrelated to codex mapper remain green; no autobyteus adapter changes required. |
| AC-006 | R-006 | With `RUN_CODEX_E2E=1`, a live `codex-runtime-graphql.e2e` case creates real local PNG fixtures, sends one as `/rest/files/...` URL and one as absolute filesystem path in `userInput.contextFiles`, and confirms Codex thread history includes both image inputs for that turn. |

## Constraints / Dependencies

- Codex app-server image input shape follows official docs: `text`, `image`, `localImage`.
- Existing GraphQL input `contextFiles` shape is unchanged.
- No backward compatibility wrappers or dual-path legacy retention.

## Assumptions

- Uploaded file URLs are generated by this server and map to local media root.
- Codex process has filesystem access to the same media root path on the server host.

## Open Questions / Risks

- If deployment topology separates web upload storage and codex process filesystem, localImage conversion may need shared volume guarantees.
- Live Codex E2E runtime behavior may vary by model/tool availability; test must avoid tool-dependent assertions and focus on user-turn image input persistence.

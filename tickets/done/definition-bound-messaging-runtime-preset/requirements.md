# Requirements

- Status: Design-ready

## Goal / Problem Statement

Current messaging bindings target active run IDs. That forces the user to start an agent runtime manually before binding, and it makes the binding fragile because it depends on an ephemeral runtime identity instead of a stable product concept. The goal is to let the user bind a messaging route directly to a specific agent definition plus a saved launch preset, then message that agent from Telegram or another managed provider without manual pre-start.

## In-Scope Use Cases

- `UC-001` Create a messaging binding to a specific agent definition in the app without requiring a live run.
- `UC-002` Configure the launch preset during binding using normal app selectors, not chat commands.
- `UC-003` When an inbound message arrives for a bound peer and no runtime is active, auto-start the agent from the saved preset and deliver the message.
- `UC-004` When a runtime already exists for that binding, reuse it for subsequent inbound messages.
- `UC-005` Keep the flow provider-generic for managed `TELEGRAM`, `DISCORD`, `WHATSAPP`, and `WECOM`.

## Out of Scope For V1

- `OOS-001` Team-definition auto-start bindings.
- `OOS-002` Chat-driven model selection, workspace selection, or raw runtime configuration.
- `OOS-003` Peer allowlist / owner-only enforcement.
- `OOS-004` `/start`, `/stop`, `/listagents`, or other chat command control flows.

## Requirements

- `REQ-001` Binding target model
  - A channel binding must target an `AGENT` definition plus a persisted launch preset, not an active run ID.

- `REQ-002` Launch preset contents
  - The launch preset must capture the fields required to start an agent runtime for messaging use.
  - Minimum fields: `agentDefinitionId`, `workspace`, `llmModelIdentifier`, `runtimeKind`, `autoExecuteTools`, `skillAccessMode`, and optional `llmConfig`.

- `REQ-003` Workspace configuration UX
  - The binding UI must let the user either select an existing workspace or enter a workspace path to create/load one.
  - The user must not need to type workspace information in chat.

- `REQ-004` Model/runtime configuration UX
  - The binding UI must let the user choose model and runtime using existing app selection patterns.
  - The user must not need to type raw model identifiers in chat.

- `REQ-005` Auto-start on inbound message
  - If no live runtime exists for a bound agent definition when an inbound message arrives, the server must start one from the binding preset before dispatch.

- `REQ-006` Runtime reuse
  - If a live runtime already exists for the binding, the server must reuse that runtime for subsequent inbound messages rather than creating a new run each time.

- `REQ-007` Provider-generic routing
  - The definition-bound preset mechanism must work across all managed non-WeChat providers supported by the messaging UI.

- `REQ-008` Outbound reply continuity
  - Replies generated by the auto-started or reused runtime must continue to flow back through the existing external-channel callback path to the originating provider peer.

- `REQ-009` Settings visibility
  - The binding setup UI must show the selected target agent definition and launch preset information clearly enough that the user can understand what will start when a message arrives.

## Acceptance Criteria

- `AC-001`
  - Given a configured managed provider account and a discovered peer,
  - when the user creates a binding to an agent definition with a valid launch preset,
  - then saving the binding succeeds without requiring any active run ID.

- `AC-002`
  - Given a valid definition-bound binding and no active runtime,
  - when an inbound message arrives for the bound peer,
  - then the server creates a runtime from the saved preset and routes the inbound message into that runtime.

- `AC-003`
  - Given a valid definition-bound binding and an already-active runtime for that binding,
  - when another inbound message arrives,
  - then the message is routed to the existing runtime instead of creating a duplicate runtime.

- `AC-004`
  - Given a runtime started from a definition-bound binding,
  - when the agent produces an outbound reply,
  - then the reply is delivered back to the messaging provider peer.

- `AC-005`
  - Given the user is configuring a binding from the settings UI,
  - when choosing the target and preset,
  - then the user can select agent, workspace, model, and runtime from the app UI without using chat commands or raw model IDs in chat.

- `AC-006`
  - Given a managed provider other than `WECHAT`,
  - when it reaches the binding step,
  - then the same definition-bound launch-preset behavior is available.

## Constraints / Dependencies

- The implementation must remain server-managed; the gateway should not own runtime startup logic.
- The transport abstraction must remain provider-generic.
- Existing runtime creation services should be reused rather than duplicated through ad hoc startup code.
- Source code changes remain locked until workflow Stage 6.

## Assumptions

- V1 may scope to `AGENT` targets only even though the broader messaging transport layer remains provider-generic.
- The existing run-history manifest shape is a valid starting point for the launch preset schema.
- Workspace identity may need to be resolved by root path at runtime instead of persisting only a workspace ID.

## Open Questions / Risks

- Whether existing run-bound bindings should be migrated automatically or invalidated for one-time reconfiguration.
- Whether v1 needs explicit runtime idle/stop policy controls or can start with simple reuse-while-running semantics.
- Whether peer allowlist / owner-only policy should become part of this change or remain separate.

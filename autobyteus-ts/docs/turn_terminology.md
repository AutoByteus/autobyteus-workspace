# Turn Terminology

This project uses two different runtime concepts that previously both used the word "turn":

1. `AgentTurn` / `turnId` (outer durable agent turn)
- Scope: one user-originated interaction cycle.
- Source: generated by `MemoryManager.startTurn()` and stored in `AgentRuntimeState.activeTurn`.
- Usage:
  - owns the active outer runtime turn for user input, streaming, tool loops, and assistant completion
  - supplies `turnId` to runtime events and tool invocations
  - supplies `turn_id` to streamed segment events and tool lifecycle payloads

2. `ToolInvocationBatch` (tool aggregation batch)
- Scope: one set of tool calls emitted by a single LLM response.
- Source: created in `LLMUserMessageReadyEventHandler` when tool invocations are parsed from streamed output.
- Usage: temporarily collects tool results and gates when to enqueue one aggregated tool-results message back into the input pipeline.

## Relationship

- One outer `AgentTurn` (`turnId`) can contain zero, one, or multiple `ToolInvocationBatch` instances.
- `turnId` is the durable runtime outer-turn identity.
- `ToolInvocationBatch` is short-lived orchestration state.
- Persisted memory traces still serialize the low-level storage field as `turn_id`; that matches the canonical payload/storage naming for the same turn identity.

## Cross-repo note

- Touched `autobyteus-server` consumers stay on `turn_id` / `turnId`; this refactor keeps one canonical turn vocabulary across repos.
- Memory view APIs still expose the persisted trace turn (`turn_id` / `turnId`) because the storage schema did not change in this refactor.
- `autobyteus-server` and `autobyteus-web` do not expose the transient `ToolInvocationBatch` orchestration state.
